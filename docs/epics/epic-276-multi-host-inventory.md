# Epic 276: Multi-Host Ansible Inventory & Service Discovery

**Status**: PENDING
**Priority**: High
**Depends on**: None
**Blocks**: Epic 277, Epic 278, Epic 279, Epic 280
**Branch**: `desktop`

## Summary

Transform the current single-host Ansible deployment into a matrix-docker-ansible-deploy style architecture where each service (app, postgres, caddy, minio, strfry, transcription, asterisk, signal) is a toggleable role with its own Docker Compose file. Multi-host inventory groups allow services to be distributed across machines, with automatic service discovery wiring cross-host connections.

## Problem Statement

The current `deploy/ansible/` assumes all services run on a single host via one monolithic `docker-compose.j2` template. This creates three problems:

1. **No horizontal scaling.** Operators running 10-50 hubs cannot split PostgreSQL onto a dedicated database server, or run the Nostr relay on a separate box with different resource constraints.
2. **All-or-nothing configuration.** The `compose_profiles` mechanism (transcription, asterisk, signal) uses Jinja2 conditionals inside a single template, making it fragile and hard to extend with new services.
3. **No cross-host service discovery.** When postgres runs on a different machine than the app, connection strings must be manually constructed. The matrix-docker-ansible-deploy model solves this with inventory-driven variable injection.

The reference model (matrix-docker-ansible-deploy) serves 1000+ deployments with one inventory per deployment, N hosts, and every service toggleable. Llamenos needs the same pattern for its target of 1-50 hubs managed by 2-3 part-time operators.

## Implementation

### Phase 1: Inventory Groups & Service Toggle Variables

Replace the flat `all.hosts.llamenos` inventory with group-based inventory:

**File: `deploy/ansible/inventory.example.yml`**

```yaml
# Llamenos Multi-Host Inventory
#
# Single-host (everything on one server — default):
#   Put your server in [llamenos_servers] and all services deploy there.
#
# Multi-host (split services across machines):
#   Add hosts to specific groups. A host can be in multiple groups.
#   Services deploy only to hosts in their group.

all:
  vars:
    # Global variables shared across all hosts
    ansible_python_interpreter: /usr/bin/python3

  children:
    # Every Llamenos host must be in this group
    llamenos_servers:
      hosts:
        hub1:
          ansible_host: 203.0.113.10
          ansible_user: deploy
          ansible_ssh_private_key_file: ~/.ssh/llamenos_deploy
          ansible_port: 22

    # Service groups — assign hosts to determine where each service runs.
    # If a group is empty or undefined, that service runs on all llamenos_servers hosts.
    llamenos_app:
      hosts:
        hub1: {}
    llamenos_db:
      hosts:
        hub1: {}
    llamenos_proxy:
      hosts:
        hub1: {}
    llamenos_storage:
      hosts:
        hub1: {}
    llamenos_relay:
      hosts:
        hub1: {}
    # Optional service groups — only used if the service is enabled
    llamenos_transcription:
      hosts: {}
    llamenos_asterisk:
      hosts: {}
    llamenos_signal:
      hosts: {}
```

      - /tmp:size=64M

### Phase 3: Service Discovery via Ansible Variables

Create a role `service-discovery` that runs before all service roles and computes connection variables based on inventory topology.

**File: `deploy/ansible/roles/service-discovery/tasks/main.yml`**

```yaml
---
# Service Discovery Role
#
# Computes cross-host connection variables based on inventory groups.
# When a service (e.g., postgres) runs on a different host than the consumer
# (e.g., app), the consumer gets the remote host's IP as the connection target.
#
# Single-host: all services use Docker network names (e.g., "postgres:5432").
# Multi-host: remote services use ansible_host IPs.

- name: Determine PostgreSQL connection target
  ansible.builtin.set_fact:
    llamenos_pg_host: >-
      {%- if groups.get('llamenos_db', []) | length > 0 and
             inventory_hostname not in groups.get('llamenos_db', []) -%}
        {{ hostvars[groups['llamenos_db'][0]]['ansible_host'] }}
      {%- else -%}
        postgres
      {%- endif -%}
    llamenos_pg_port: >-
      {%- if groups.get('llamenos_db', []) | length > 0 and
             inventory_hostname not in groups.get('llamenos_db', []) -%}
        {{ llamenos_postgres_port | default('5432') }}
      {%- else -%}
        5432
      {%- endif -%}

- name: Determine MinIO connection target
  ansible.builtin.set_fact:
    llamenos_minio_endpoint: >-
      {%- if groups.get('llamenos_storage', []) | length > 0 and
             inventory_hostname not in groups.get('llamenos_storage', []) -%}
        http://{{ hostvars[groups['llamenos_storage'][0]]['ansible_host'] }}:9000
      {%- else -%}
        http://minio:9000
      {%- endif -%}

- name: Determine strfry relay connection target
  ansible.builtin.set_fact:
    llamenos_relay_url: >-
      {%- if groups.get('llamenos_relay', []) | length > 0 and
             inventory_hostname not in groups.get('llamenos_relay', []) -%}
        ws://{{ hostvars[groups['llamenos_relay'][0]]['ansible_host'] }}:7777
      {%- else -%}
        ws://strfry:7777
      {%- endif -%}

- name: Determine Whisper transcription URL
  ansible.builtin.set_fact:
    llamenos_whisper_url: >-
      {%- if groups.get('llamenos_transcription', []) | length > 0 and
             inventory_hostname not in groups.get('llamenos_transcription', []) -%}
        http://{{ hostvars[groups['llamenos_transcription'][0]]['ansible_host'] }}:8080/v1/audio/transcriptions
      {%- else -%}
        http://whisper:8080/v1/audio/transcriptions
      {%- endif -%}

- name: Compute DATABASE_URL for app service
  ansible.builtin.set_fact:
    llamenos_database_url: "postgresql://llamenos:{{ pg_password }}@{{ llamenos_pg_host }}:{{ llamenos_pg_port }}/llamenos"

- name: Display service topology
  ansible.builtin.debug:
    msg: |
      Service topology for {{ inventory_hostname }}:
        PostgreSQL: {{ llamenos_pg_host }}:{{ llamenos_pg_port }}
        MinIO: {{ llamenos_minio_endpoint }}
        Relay: {{ llamenos_relay_url }}
        Whisper: {{ llamenos_whisper_url | default('disabled') }}
```

### Phase 4: Per-Service Ansible Roles

Refactor the monolithic `roles/llamenos/` into per-service roles. Each role:
- Checks if the service is enabled (`llamenos_{service}_enabled`)
- Checks if the current host is in the service's inventory group
- Templates its compose file and env file
- Starts its container

**File: `deploy/ansible/roles/llamenos-app/tasks/main.yml`**

```yaml
---
# App service role — deploys the Node.js application container

- name: Skip if app not enabled or not assigned to this host
  ansible.builtin.meta: end_play
  when: >
    not llamenos_app_enabled or
    (groups.get('llamenos_app', []) | length > 0 and
     inventory_hostname not in groups.get('llamenos_app', []))

- name: Create app directory
  ansible.builtin.file:
    path: "{{ app_dir }}/services/app"
    state: directory
    owner: "{{ deploy_user }}"
    group: "{{ deploy_group }}"
    mode: "0750"

- name: Template app compose file
  ansible.builtin.template:
    src: compose/app.j2
    dest: "{{ app_dir }}/services/app/docker-compose.yml"
    owner: "{{ deploy_user }}"
    group: "{{ deploy_group }}"
    mode: "0640"
  notify: Restart app service

- name: Template app environment
  ansible.builtin.template:
    src: env/app.j2
    dest: "{{ app_dir }}/services/app/.env.app"
    owner: "{{ deploy_user }}"
    group: "{{ deploy_group }}"
    mode: "0600"
  notify: Restart app service

- name: Start app service
  community.docker.docker_compose_v2:
    project_src: "{{ app_dir }}/services/app"
    state: present
  register: app_compose

- name: Wait for app health check
  ansible.builtin.uri:
    url: "http://localhost:3000/api/health"
    method: GET
    status_code: 200
    return_content: true
  register: app_health
  retries: 20
  delay: 5
  until: app_health.status == 200
```

### Phase 5: Orchestration Playbook

Create a new `deploy.yml` that replaces the current one, running roles in dependency order:

**File: `deploy/ansible/playbooks/deploy.yml`** (rewritten):

```yaml
---
# Llamenos Deployment — Multi-Host Orchestration
#
# Deploys services in dependency order across all hosts.
# Each service role is self-selecting: it skips hosts not in its group.

- name: Gather facts and run service discovery
  hosts: llamenos_servers
  become: true
  vars_files:
    - ../vars.yml
  roles:
    - role: service-discovery
      tags: [always]

- name: Deploy infrastructure services (database, storage, relay)
  hosts: llamenos_servers
  become: true
  vars_files:
    - ../vars.yml
  roles:
    - role: docker
      tags: [docker, deploy]
    - role: llamenos-postgres
      tags: [postgres, deploy]
      when: llamenos_postgres_enabled | default(true)
    - role: llamenos-minio
      tags: [minio, deploy]
      when: llamenos_minio_enabled | default(true)
    - role: llamenos-strfry
      tags: [strfry, deploy]
      when: llamenos_strfry_enabled | default(true)

- name: Deploy application services
  hosts: llamenos_servers
  become: true
  vars_files:
    - ../vars.yml
  roles:
    - role: llamenos-app
      tags: [app, deploy]
      when: llamenos_app_enabled | default(true)

- name: Deploy proxy and optional services
  hosts: llamenos_servers
  become: true
  vars_files:
    - ../vars.yml
  roles:
    - role: llamenos-caddy
      tags: [caddy, deploy]
      when: llamenos_caddy_enabled | default(true)
    - role: llamenos-whisper
      tags: [whisper, deploy]
      when: llamenos_transcription_enabled | default(false)
    - role: llamenos-asterisk
      tags: [asterisk, deploy]
      when: llamenos_asterisk_enabled | default(false)
    - role: llamenos-signal
      tags: [signal, deploy]
      when: llamenos_signal_enabled | default(false)
```

### Phase 6: Internal TLS (Optional)

When `llamenos_internal_tls_enabled: true`, generate a self-signed CA and per-host certificates for cross-host service communication.

**File: `deploy/ansible/roles/internal-tls/tasks/main.yml`**

```yaml
---
# Internal TLS role
# Generates a deployment-local CA and per-host certificates for
# cross-host service communication (PostgreSQL, MinIO, strfry).

- name: Create TLS directory
  ansible.builtin.file:
    path: "{{ app_dir }}/tls"
    state: directory
    owner: "{{ deploy_user }}"
    group: "{{ deploy_group }}"
    mode: "0700"

- name: Generate CA private key (on first host only)
  community.crypto.openssl_privatekey:
    path: "{{ app_dir }}/tls/ca.key"
    size: 4096
  when: inventory_hostname == groups['llamenos_servers'][0]
  run_once: true

- name: Generate CA certificate
  community.crypto.x509_certificate:
    path: "{{ app_dir }}/tls/ca.crt"
    privatekey_path: "{{ app_dir }}/tls/ca.key"
    provider: selfsigned
    selfsigned_not_after: "+3650d"
  when: inventory_hostname == groups['llamenos_servers'][0]
  run_once: true

- name: Distribute CA certificate to all hosts
  ansible.builtin.copy:
    src: "{{ app_dir }}/tls/ca.crt"
    dest: "{{ app_dir }}/tls/ca.crt"
    owner: "{{ deploy_user }}"
    group: "{{ deploy_group }}"
    mode: "0644"

- name: Generate per-host private key
  community.crypto.openssl_privatekey:
    path: "{{ app_dir }}/tls/{{ inventory_hostname }}.key"
    size: 2048

- name: Generate per-host CSR
  community.crypto.openssl_csr:
    path: "{{ app_dir }}/tls/{{ inventory_hostname }}.csr"
    privatekey_path: "{{ app_dir }}/tls/{{ inventory_hostname }}.key"
    subject_alt_name:
      - "DNS:{{ inventory_hostname }}"
      - "IP:{{ ansible_host }}"

- name: Sign per-host certificate with CA
  community.crypto.x509_certificate:
    path: "{{ app_dir }}/tls/{{ inventory_hostname }}.crt"
    csr_path: "{{ app_dir }}/tls/{{ inventory_hostname }}.csr"
    ownca_path: "{{ app_dir }}/tls/ca.crt"
    ownca_privatekey_path: "{{ app_dir }}/tls/ca.key"
    provider: ownca
    ownca_not_after: "+365d"
```

### Phase 7: Migration & Backward Compatibility

- Existing single-host deployments continue working without inventory changes (all services default to `llamenos_servers` group)
- Old `templates/docker-compose.j2` is kept but marked deprecated with a comment pointing to the new per-service templates
- `just` commands remain unchanged; `just deploy` calls the new playbook transparently
- A migration playbook detects old-style deployments and restructures the `app_dir` layout

**File: `deploy/ansible/playbooks/migrate-to-multi-host.yml`**

```yaml
---
# Migration: single-host monolithic → multi-host per-service layout
#
# Safe to run multiple times (idempotent).
# Stops the old monolithic stack, moves data dirs, starts per-service stacks.

- name: Migrate to multi-host layout
  hosts: llamenos_servers
  become: true
  vars_files:
    - ../vars.yml
  tasks:
    - name: Check for old monolithic docker-compose.yml
      ansible.builtin.stat:
        path: "{{ app_dir }}/docker-compose.yml"
      register: old_compose

    - name: Stop old monolithic stack
      community.docker.docker_compose_v2:
        project_src: "{{ app_dir }}"
        state: stopped
      when: old_compose.stat.exists

    - name: Create per-service directory structure
      ansible.builtin.file:
        path: "{{ app_dir }}/services/{{ item }}"
        state: directory
        owner: "{{ deploy_user }}"
        group: "{{ deploy_group }}"
        mode: "0750"
      loop:
        - app
        - postgres
        - caddy
        - minio
        - strfry

    - name: Archive old compose file
      ansible.builtin.copy:
        src: "{{ app_dir }}/docker-compose.yml"
        dest: "{{ app_dir }}/docker-compose.yml.pre-migration"
        remote_src: true
      when: old_compose.stat.exists

    - name: Remove old compose file
      ansible.builtin.file:
        path: "{{ app_dir }}/docker-compose.yml"
        state: absent
      when: old_compose.stat.exists

    - name: Display migration status
      ansible.builtin.debug:
        msg: |
          Migration complete. Run 'just deploy' to start per-service stacks.
          Old compose file archived at: {{ app_dir }}/docker-compose.yml.pre-migration
```

## Testing

1. **Single-host smoke test**: Deploy with unchanged single-host inventory. All services must start on one machine, health checks pass. Verify `docker compose ps` shows all enabled services running.

2. **Multi-host topology test**: Use 2-host inventory (app+caddy on host A, postgres+minio+strfry on host B). Verify:
   - App connects to PostgreSQL via remote IP
   - App connects to MinIO via remote IP
   - App connects to strfry via remote IP
   - Health check passes end-to-end

3. **Service toggle test**: Set `llamenos_transcription_enabled: false`, verify Whisper container is not deployed. Set to `true`, re-run, verify it appears.

4. **Migration test**: Start with current monolithic deployment, run `migrate-to-multi-host.yml`, then `deploy.yml`. Verify data volumes are preserved and all services come back up.

5. **Internal TLS test**: Enable `llamenos_internal_tls_enabled: true` on multi-host setup. Verify PostgreSQL accepts connections with client certificates.

6. **Idempotency test**: Run `just deploy` twice in succession. Second run should report zero changes.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Breaking existing single-host deployments | Medium | High | Backward-compatible defaults; migration playbook tested before release; old compose template kept as fallback |
| Docker network isolation loss in multi-host | Low | High | Cross-host services use firewall rules (UFW) to restrict ports to known peer IPs; internal TLS optional but documented |
| Service discovery produces wrong connection strings | Medium | Medium | Integration test with 2-host topology; debug output in service-discovery role |
| Named volume conflicts during migration | Low | High | Migration playbook archives old compose file, does not touch volumes; Docker named volumes persist across compose project renames |
| Complexity overwhelming part-time operators | Medium | Medium | Single-host remains the default and documented path; multi-host is opt-in for advanced deployments |
