# Epic 214: Mobile Crypto Integration (UniFFI Linking)

## Goal

Replace stand-in crypto implementations on iOS and Android with real llamenos-core Rust crypto via UniFFI bindings. This is the single most critical pre-launch task — without it, mobile E2EE is simulated, not real.

## Context

Both iOS and Android apps currently use stand-in implementations:
- **iOS**: `#if canImport(LlamenosCore)` guards with SHA256-based mock encryption
- **Android**: `nativeLibLoaded = false` check with mock crypto

The Rust crate at `packages/crypto/` already supports UniFFI (`--features mobile`). The build infrastructure for mobile targets exists at `packages/crypto/scripts/build-mobile.sh`.

## Implementation

### 1. Build Script Updates

Verify `packages/crypto/scripts/build-mobile.sh` produces:

**iOS**:
- `LlamenosCoreFFI.xcframework` (arm64 device + arm64 simulator)
- Swift bindings generated by `uniffi-bindgen-swift`

**Android**:
- JNI `.so` files for `arm64-v8a`, `armeabi-v7a`, `x86_64`, `x86`
- Kotlin bindings generated by `uniffi-bindgen-kotlin`

### 2. iOS Integration

1. Build XCFramework: `cd packages/crypto && ./scripts/build-mobile.sh ios`
2. Copy to `apps/ios/LlamenosCoreFFI.xcframework/` (gitignored)
3. Add to Package.swift as a binary target:

```swift
let package = Package(
    name: "Llamenos",
    platforms: [.iOS(.v17)],
    dependencies: [],
    targets: [
        .binaryTarget(
            name: "LlamenosCoreFFI",
            path: "LlamenosCoreFFI.xcframework"
        ),
        .target(
            name: "Llamenos",
            dependencies: ["LlamenosCoreFFI"],
            path: "Sources"
        ),
        .testTarget(
            name: "LlamenosTests",
            dependencies: ["Llamenos"],
            path: "Tests"
        ),
    ]
)
```

4. The `#if canImport(LlamenosCore)` guards already handle the switch
5. Verify all crypto tests pass against real implementation

### 3. Android Integration

1. Build JNI libs: `cd packages/crypto && ./scripts/build-mobile.sh android`
2. Copy `.so` files to `apps/android/app/src/main/jniLibs/{abi}/`
3. Copy generated Kotlin bindings to source tree
4. The `nativeLibLoaded` check already handles the switch
5. Verify all crypto tests pass against real implementation

### 4. CI Integration

Add crypto build steps to `mobile-release.yml`:

```yaml
- name: Build crypto for Android
  run: cd packages/crypto && ./scripts/build-mobile.sh android

- name: Build crypto for iOS
  run: cd packages/crypto && ./scripts/build-mobile.sh ios
```

### 5. Cross-Platform Test Verification

After linking real crypto on all platforms, run cross-platform test vectors (from Epic 212) to verify identical outputs for:
- Schnorr signature generation/verification
- ECIES encrypt/decrypt roundtrip
- PBKDF2 PIN key derivation
- HKDF domain-separated key derivation
- XChaCha20-Poly1305 symmetric encryption
- SAS code derivation from shared secrets

## Verification

1. `swift build` succeeds with LlamenosCoreFFI linked
2. `./gradlew assembleDebug` succeeds with JNI libs present
3. All mobile crypto unit tests pass against real implementation
4. Cross-platform test vectors produce identical results
5. E2EE note created on desktop decrypts on mobile (and vice versa)
6. PIN encryption/decryption works identically across platforms
7. Device linking ECDH produces correct shared secret across platforms

## Dependencies

- Epic 212 (Test Coverage) — cross-platform test vectors

## Risk

- **High**: UniFFI-generated bindings may have ABI differences between Rust and host language
- **Medium**: XCFramework code signing for iOS distribution
- **Low**: JNI loading on different Android architectures
