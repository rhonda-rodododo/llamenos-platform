# Epic 207: Android Client Foundation

## Goal

Build the native Kotlin/Compose Android app with crypto integration via UniFFI/JNI, secure key storage via Android Keystore, and the core auth flow (login, onboarding, PIN). Mirrors Epic 206 (iOS) with Android-native patterns.

## Context

Same rationale as Epic 206 — React Native + Detox has fundamental issues. Native Android with Espresso/Compose testing avoids them entirely. The app uses the same Rust crypto core via JNI (UniFFI-generated Kotlin bindings).

### Architecture Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| UI Framework | Jetpack Compose | Modern declarative UI, matches SwiftUI mental model |
| Min SDK | 26 (Android 8.0) | Covers 97%+ of active devices, Keystore attestation |
| Dependency Mgmt | Gradle + Version Catalogs | Standard Android, type-safe dependency management |
| Architecture | MVVM + Hilt DI | Android-standard, testable, lifecycle-aware |
| Crypto | UniFFI/JNI from `packages/crypto/` | Same Rust code as desktop and iOS |
| Key Storage | Android Keystore + EncryptedSharedPreferences | Hardware-backed where available |
| Networking | OkHttp + Kotlin Coroutines | Industry standard, built-in retry/interceptors |
| VoIP | Twilio Voice Android SDK | Matches iOS SDK, ConnectionService integration |

### Buildit Android Patterns (Selective Adoption)

Buildit's Android client uses:
- **YES**: Hilt DI, Compose Navigation with type-safe routes, CryptoManager singleton
- **YES**: JNI library loading with fallback, `withContext(Dispatchers.Default)` for crypto
- **NO**: Room + SQLCipher (no local DB in llamenos)
- **NO**: Clean Architecture with data/domain/presentation layers (overkill for our simpler app)
- **NO**: Module registry system (llamenos has fewer features)

## CryptoService Design

Same principle as iOS: nsec NEVER leaves this class. All crypto happens through delegated calls.

```kotlin
@Singleton
class CryptoService @Inject constructor() {
    private var nsecHex: String? = null
    var pubkey: String? = null
        private set
    var npub: String? = null
        private set

    val isUnlocked: Boolean get() = nsecHex != null

    init {
        // Load native library
        try {
            System.loadLibrary("llamenos_core")
        } catch (e: UnsatisfiedLinkError) {
            throw IllegalStateException("Failed to load llamenos-core native library", e)
        }
    }

    fun generateKeypair(): Pair<String, String> {
        val kp = LlamenosCore.generateKeypair()
        nsecHex = kp.secretKeyHex
        pubkey = kp.publicKey
        npub = kp.npub
        return Pair(kp.nsec, kp.npub)
    }

    fun importNsec(nsec: String) {
        val kp = LlamenosCore.keypairFromNsec(nsec)
        nsecHex = kp.secretKeyHex
        pubkey = kp.publicKey
        npub = kp.npub
    }

    suspend fun encryptForStorage(pin: String): EncryptedKeyData = withContext(Dispatchers.Default) {
        val nsec = nsecHex ?: throw CryptoException("No key loaded")
        LlamenosCore.encryptWithPin(nsec = nsec, pin = pin, pubkeyHex = pubkey!!)
    }

    suspend fun decryptFromStorage(data: EncryptedKeyData, pin: String) = withContext(Dispatchers.Default) {
        val nsec = LlamenosCore.decryptWithPin(data = data, pin = pin)
        importNsec(nsec)
    }

    suspend fun createAuthToken(method: String, path: String): AuthToken = withContext(Dispatchers.Default) {
        val nsec = nsecHex ?: throw CryptoException("No key loaded")
        val timestamp = System.currentTimeMillis()
        LlamenosCore.createAuthToken(
            secretKeyHex = nsec,
            timestamp = timestamp,
            method = method,
            path = path
        )
    }

    suspend fun encryptNote(payload: String, adminPubkeys: List<String>): EncryptedNote =
        withContext(Dispatchers.Default) {
            val pk = pubkey ?: throw CryptoException("No key loaded")
            LlamenosCore.encryptNoteForRecipients(
                payloadJson = payload,
                authorPubkey = pk,
                adminPubkeys = adminPubkeys
            )
        }

    fun lock() {
        nsecHex = null
        // pubkey and npub remain for display
    }
}
```

### Key Differences from iOS

- **Dispatchers.Default** for crypto ops — avoids blocking main thread (Android ANR)
- **Hilt @Singleton** — DI-managed lifecycle
- **System.loadLibrary** — JNI loading at init time, fail fast

## Keystore Integration

```kotlin
@Singleton
class KeystoreService @Inject constructor(
    @ApplicationContext private val context: Context
) {
    private val prefs: SharedPreferences by lazy {
        EncryptedSharedPreferences.create(
            "llamenos_secure_prefs",
            MasterKey.Builder(context)
                .setKeyScheme(MasterKey.KeyScheme.AES256_GCM)
                .build(),
            context,
            EncryptedSharedPreferences.PrefKeyEncryptionScheme.AES256_SIV,
            EncryptedSharedPreferences.PrefValueEncryptionScheme.AES256_GCM
        )
    }

    fun store(key: String, value: String) {
        prefs.edit().putString(key, value).apply()
    }

    fun retrieve(key: String): String? {
        return prefs.getString(key, null)
    }

    fun delete(key: String) {
        prefs.edit().remove(key).apply()
    }

    fun clear() {
        prefs.edit().clear().apply()
    }
}
```

- `"encrypted-keys"` — PIN-encrypted nsec JSON

```kotlin
                row.forEach { label ->

                        Spacer(Modifier.size(72.dp))

                            shape = CircleShape
                        ) {
                            Text(
                                text = label,
                                style = MaterialTheme.typography.headlineMedium
                            )
                        }
                    }
                }
            }
        }
    }
```

## API Service with Auth Interceptor

```kotlin
@Singleton
class ApiService @Inject constructor(
    private val cryptoService: CryptoService
) {
    private val client = OkHttpClient.Builder()
        .addInterceptor(AuthInterceptor(cryptoService))
        .build()

    private val json = Json { ignoreUnknownKeys = true }

    suspend inline fun <reified T> request(
        method: String,
        path: String,
        body: Any? = null
    ): T = withContext(Dispatchers.IO) {
        val url = "${getBaseUrl()}$path"
        val requestBody = body?.let {
            json.encodeToString(it).toRequestBody("application/json".toMediaType())
        }

        val request = Request.Builder()
            .url(url)
            .method(method, requestBody)
            .build()

        val response = client.newCall(request).execute()
        if (!response.isSuccessful) throw ApiException(response.code, response.message)

        json.decodeFromString(response.body!!.string())
    }
}

class AuthInterceptor(private val cryptoService: CryptoService) : Interceptor {
    override fun intercept(chain: Interceptor.Chain): Response {
        val request = chain.request()
        val path = request.url.encodedPath

        // Build auth header synchronously — createAuthToken is fast (Schnorr signing).
        // We use synchronized to prevent a race between isUnlocked check and token
        // creation if lock() is called concurrently (e.g., background timeout).
        val authRequest = synchronized(cryptoService) {
            if (cryptoService.isUnlocked) {
                try {
                    val token = cryptoService.createAuthTokenSync(request.method, path)
                    val authHeader = """{"pubkey":"${token.pubkey}","timestamp":${token.timestamp},"token":"${token.token}"}"""
                    request.newBuilder()
                        .header("Authorization", "Bearer $authHeader")
                        .build()
                } catch (e: Exception) {
                    null  // Key was locked between check and sign — proceed unauthenticated
                }
            } else null
        }

        return chain.proceed(authRequest ?: request)
    }
}
// Note: CryptoService needs a non-suspend `createAuthTokenSync()` variant for
// the interceptor. Schnorr signing is ~1ms so blocking the OkHttp thread is fine.
```

## App Lifecycle & Security

```kotlin
@AndroidEntryPoint
class MainActivity : ComponentActivity() {
    @Inject lateinit var cryptoService: CryptoService

    private var backgroundTimestamp: Long? = null
    private val lockTimeout = 5 * 60 * 1000L  // 5 minutes

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)

        setContent {
            LlamenosTheme {
                LlamenosNavigation(cryptoService = cryptoService)
            }
        }
    }

    override fun onStop() {
        super.onStop()
        backgroundTimestamp = System.currentTimeMillis()
    }

    override fun onStart() {
        super.onStart()
        backgroundTimestamp?.let { bg ->
            if (System.currentTimeMillis() - bg > lockTimeout) {
                cryptoService.lock()
            }
        }
        backgroundTimestamp = null
    }
}
```

## Testing Strategy

### Unit Tests (JUnit)

```kotlin
class CryptoServiceTest {
    private lateinit var cryptoService: CryptoService

    @Before
    fun setup() {
        System.loadLibrary("llamenos_core")
        cryptoService = CryptoService()
    }

    @Test
    fun `generate keypair produces valid nsec and npub`() {
        val (nsec, npub) = cryptoService.generateKeypair()
        assertTrue(nsec.startsWith("nsec1"))
        assertTrue(npub.startsWith("npub1"))
        assertTrue(cryptoService.isUnlocked)
    }

    @Test
    fun `lock clears private key`() {
        cryptoService.generateKeypair()
        cryptoService.lock()
        assertFalse(cryptoService.isUnlocked)
    }

    @Test
    fun `PIN encrypt decrypt roundtrip`() = runBlocking {
        cryptoService.generateKeypair()
        val originalPubkey = cryptoService.pubkey!!

        val encrypted = cryptoService.encryptForStorage("1234")
        cryptoService.lock()

        cryptoService.decryptFromStorage(encrypted, "1234")
        assertEquals(originalPubkey, cryptoService.pubkey)
    }
}
```

## Risk Assessment

- **Low risk**: Compose + Material 3 — stable, well-documented
- **Medium risk**: JNI library loading — needs `.so` files for all ABIs, potential UnsatisfiedLinkError on older devices
- **Low risk**: EncryptedSharedPreferences — standard Jetpack library
- **Medium risk**: Hilt + Compose navigation integration — well-documented but complex setup
- **Mitigation**: Start with CryptoService unit tests (JNI validation), then build UI
