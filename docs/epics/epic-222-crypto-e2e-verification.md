# Epic 222: Crypto Interop E2E Verification

**Status: PENDING**
**Priority**: Critical — security-critical crypto must be verified on real devices
**Depends on**: Epic 218 (BDD framework), Epic 219 (auth flow for device setup)
**Blocks**: None

## Summary

Verify that the llamenos-core crypto operations work correctly on real Android hardware by running the same test vectors that the Rust and TypeScript implementations validate. This is a device-level crypto interop test: the same `test-vectors.json` fixtures are loaded on the Pixel 6a and verified against the native crypto implementation. This is distinct from the Rust `cargo test` (which runs on the host CPU) — these tests prove that the JNI/UniFFI bridge, the ARM-compiled `.so` files, and the Android runtime all work together correctly.

## Android Test Implementation

### Loading Test Vectors

Test vectors are loaded from `androidTest/assets/test-vectors.json`:

```kotlin
@HiltAndroidTest
@RunWith(AndroidJUnit4::class)
class CryptoInteropTest {
    @get:Rule val hiltRule = HiltAndroidRule(this)

    @Inject lateinit var cryptoService: CryptoService

    private lateinit var vectors: TestVectorsJson

    @Before
    fun setup() {
        hiltRule.inject()
        // Load test vectors from assets
        val context = InstrumentationRegistry.getInstrumentation().context
        val json = context.assets.open("test-vectors.json")
            .bufferedReader().readText()
        vectors = Json { ignoreUnknownKeys = true }
            .decodeFromString(json)
    }

    @Test
    fun keypairGenerationProducesValidFormat() {
        val (nsec, npub) = cryptoService.generateKeypair()
        assertTrue(nsec.startsWith("nsec1"))
        assertTrue(npub.startsWith("npub1"))
        assertEquals(63, nsec.length)
        assertEquals(63, npub.length)
    }

    @Test
    fun pinEncryptionRoundtripWithCorrectPin() = runBlocking {
        cryptoService.generateKeypair()
        val originalPubkey = cryptoService.pubkey!!

        val encrypted = cryptoService.encryptForStorage("1234")
        cryptoService.lock()
        assertFalse(cryptoService.isUnlocked)

        cryptoService.decryptFromStorage(encrypted, "1234")
        assertTrue(cryptoService.isUnlocked)
        assertEquals(originalPubkey, cryptoService.pubkey)
    }

    @Test
    fun pinEncryptionFailsWithWrongPin() = runBlocking {
        cryptoService.generateKeypair()
        val encrypted = cryptoService.encryptForStorage("1234")
        cryptoService.lock()

        try {
            cryptoService.decryptFromStorage(encrypted, "9999")
            fail("Should have thrown CryptoException")
        } catch (e: CryptoException) {
            assertFalse(cryptoService.isUnlocked)
        }
    }
}
```

### Placeholder vs Native Tests

The crypto tests must work in **two modes**:

1. **Native lib loaded** (`nativeLibLoaded = true`): Tests verify against the real Rust crypto (UniFFI). This is the primary verification path.
2. **Placeholder mode** (`nativeLibLoaded = false`): Tests verify that the placeholder implementations produce structurally valid output (correct format, correct lengths). This mode runs when the `.so` files aren't available.

```kotlin
@Test
fun noteEncryptionProducesValidStructure() = runBlocking {
    cryptoService.generateKeypair()
    val payload = """{"text":"Test","fields":null}"""
    val encrypted = cryptoService.encryptNote(payload, emptyList())

    // Valid in both native and placeholder mode
    assertTrue(encrypted.ciphertext.isNotEmpty())
    assertTrue(encrypted.envelopes.isNotEmpty())
    assertEquals(cryptoService.pubkey, encrypted.envelopes[0].recipientPubkey)
}
```
