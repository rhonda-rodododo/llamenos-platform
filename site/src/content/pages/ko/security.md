## 호스팅 제공업체가 소환장을 받는 경우

| 제공할 수 있는 것 | 제공할 수 없는 것 |
|-------------------|-------------------|
| 통화/메시지 메타데이터 (시간, 기간) | 메모 내용, 변환 기록, 신고 본문 |
| 암호화된 데이터베이스 블롭 | 자원봉사자 이름 (종단 간 암호화) |
| 언제 어떤 자원봉사자가 활동했는지 | 연락처 디렉토리 기록 (종단 간 암호화) |
| | 메시지 내용 (도착 시 암호화, 암호문으로 저장) |
| | 복호화 키 (PIN, 신원 제공업체 계정, 선택적 하드웨어 보안 키로 보호) |
| | 개별 메모 암호화 키 (임시 — 래핑 후 폐기) |
| | 전화번호 해시를 되돌리기 위한 HMAC 시크릿 |

**서버는 읽을 수 없는 데이터를 저장합니다.** 메타데이터 (언제, 얼마나, 어떤 계정)는 보입니다. 내용 (무엇을 말했는지, 무엇을 썼는지, 연락처가 누구인지)은 보이지 않습니다.

## 기능별

개인정보 노출 정도는 활성화한 채널에 따라 달라집니다:

### 음성 통화

| 사용하는 것... | 제3자가 접근 가능 | 서버가 접근 가능 | E2EE 콘텐츠 |
|----------------|-------------------|------------------|-------------|
| Twilio/SignalWire/Vonage/Plivo | 통화 오디오 (실시간), 통화 기록 | 통화 메타데이터 | 메모, 변환 기록 |
| 자체 호스팅 Asterisk | 없음 (사용자가 제어) | 통화 메타데이터 | 메모, 변환 기록 |
| 브라우저 간 (WebRTC) | 없음 | 통화 메타데이터 | 메모, 변환 기록 |

**전화 서비스 제공업체 소환장**: 통화 세부 기록 (시간, 전화번호, 기간)을 가지고 있습니다. 통화 메모나 변환 기록은 가지고 있지 않습니다. 녹음은 기본적으로 비활성화되어 있습니다.

**변환**: 변환은 기기 내 AI를 사용하여 브라우저에서 전적으로 이루어집니다. **오디오는 기기를 절대 떠나지 않습니다.** 암호화된 변환 기록만 저장됩니다.

### 텍스트 메시징

| 채널 | 제공업체 접근 | 서버 저장 | 참고 |
|------|--------------|-----------|------|
| SMS | 전화 서비스 제공업체가 모든 메시지를 읽음 | **암호화** | 제공업체가 원본 메시지를 보유 |
| WhatsApp | Meta가 모든 메시지를 읽음 | **암호화** | 제공업체가 원본 메시지를 보유 |
| Signal | Signal 네트워크는 E2EE이지만, 브리지가 도착 시 복호화 | **암호화** | SMS보다 나으나, 영지식 아님 |

**메시지는 서버에 도착하는 순간 암호화됩니다.** 서버는 암호문만 저장합니다. 전화 또는 메시징 제공업체가 원본 메시지를 여전히 보유할 수 있습니다 — 이는 해당 플랫폼의 제한이며, 우리가 변경할 수 있는 것이 아닙니다.

**메시징 제공업체 소환장**: SMS 제공업체는 전체 메시지 내용을 가지고 있습니다. Meta는 WhatsApp 내용을 가지고 있습니다. Signal 메시지는 브리지까지 E2EE이지만, 브리지(사용자 서버에서 실행)는 저장을 위해 재암호화하기 전에 복호화합니다. 모든 경우에, **서버에는 암호문만 있습니다** — 호스팅 제공업체는 메시지 내용을 읽을 수 없습니다.

### 메모, 변환 기록 및 신고서

자원봉사자가 작성한 모든 콘텐츠는 종단 간 암호화됩니다:

- 각 메모는 **고유한 랜덤 키**를 사용합니다 (전방 비밀성 — 하나의 메모가 침해되어도 다른 메모는 영향 없음)
- 키는 자원봉사자와 각 관리자를 위해 별도로 래핑됩니다
- 서버는 암호문만 저장합니다
- 복호화는 브라우저에서 발생합니다
- **사용자 정의 필드, 신고서 내용, 첨부 파일이 모두 개별적으로 암호화됩니다**

**기기 압수**: PIN **그리고** 신원 제공업체 계정 접근 권한 없이 공격자는 계산적으로 복호화가 불가능한 암호화된 블롭을 얻습니다. 하드웨어 보안 키도 사용하면, **세 가지 독립적인 요소**가 데이터를 보호합니다.

## 자원봉사자 전화번호 개인정보 보호

자원봉사자가 개인 전화로 통화를 받으면, 전화번호가 전화 서비스 제공업체에 노출됩니다.

| 시나리오 | 전화번호 노출 대상 |
|----------|-------------------|
| 자원봉사자 전화로 PSTN 통화 | 전화 서비스 제공업체, 통신사 |
| 브라우저 간 (WebRTC) | 없음 (오디오가 브라우저에 머무름) |
| 자체 호스팅 Asterisk + SIP 전화 | 사용자의 Asterisk 서버만 |

**자원봉사자 전화번호를 보호하려면**: 브라우저 기반 통화(WebRTC)를 사용하거나 자체 호스팅 Asterisk에 연결된 SIP 전화를 제공하세요.

## 최근 출시

다음 개선 사항이 현재 이용 가능합니다:

| 기능 | 개인정보 보호 이점 |
|------|-------------------|
| 암호화된 메시지 저장소 | SMS, WhatsApp, Signal 메시지가 서버에 암호문으로 저장 |
| 기기 내 변환 | 오디오가 브라우저를 절대 떠나지 않음 — 기기에서 전적으로 처리 |
| 다중 요소 키 보호 | 암호화 키가 PIN, 신원 제공업체, 선택적 하드웨어 보안 키로 보호 |
| 하드웨어 보안 키 | 물리적 키가 원격으로 침해할 수 없는 세 번째 요소를 추가 |
| 재현 가능한 빌드 | 배포된 코드가 공개 소스와 일치하는지 확인 |
| 암호화된 연락처 디렉토리 | 연락처 기록, 관계, 메모가 종단 간 암호화 |

| 변환 기록 | 예 (E2EE) | 아니오 | 암호문만 |
| 신고서 | 예 (E2EE) | 아니오 | 암호문만 |
| 첨부 파일 | 예 (E2EE) | 아니오 | 암호문만 |
| 연락처 기록 | 예 (E2EE) | 아니오 | 암호문만 |

| 팀/역할 메타데이터 | 예 (암호화) | 아니오 | 암호문만 |
