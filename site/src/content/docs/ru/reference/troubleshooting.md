Это руководство охватывает распространённые проблемы и их решения для всех режимов развёртывания и платформ Llamenos.

## Проблемы с развёртыванием Docker

### Контейнеры не запускаются

**Отсутствующие переменные окружения:**

Docker Compose проверяет все сервисы при запуске, даже профилированные. Если вы видите ошибки об отсутствующих переменных, убедитесь, что ваш файл `.env` содержит все необходимые значения:

```bash
# Required in .env for Docker Compose
PG_PASSWORD=your_postgres_password
MINIO_ACCESS_KEY=your_minio_access_key
MINIO_SECRET_KEY=your_minio_secret_key
HMAC_SECRET=your_hmac_secret
ARI_PASSWORD=your_ari_password       # Required even if not using Asterisk
BRIDGE_SECRET=your_bridge_secret     # Required even if not using Asterisk
ADMIN_PUBKEY=your_admin_hex_pubkey
```

Даже если вы не используете мост Asterisk, Docker Compose проверяет его определение сервиса и требует наличия `ARI_PASSWORD` и `BRIDGE_SECRET`.

**Конфликты портов:**

Если порт уже занят, проверьте, какой процесс его использует:

```bash
# Check what's using port 8787 (Worker)
sudo lsof -i :8787

# Check what's using port 5432 (PostgreSQL)
sudo lsof -i :5432

# Check what's using port 9000 (MinIO)
sudo lsof -i :9000
```

Остановите конфликтующий процесс или измените сопоставление портов в `docker-compose.yml`.

### Ошибки подключения к базе данных

Если приложение не может подключиться к PostgreSQL:

- Убедитесь, что `PG_PASSWORD` в `.env` совпадает с тем, что использовалось при первом создании контейнера
- Убедитесь, что контейнер PostgreSQL работает исправно: `docker compose ps`
- Если пароль был изменён, возможно, потребуется удалить том и пересоздать: `docker compose down -v && docker compose up -d`

### Релей Strfry не подключается

Nostr-релей (strfry) является основным сервисом, а не опциональным. Если релей не запущен:

```bash
# Check relay status
docker compose logs strfry

# Restart the relay
docker compose restart strfry
```

Если релей не запускается, проверьте конфликты на порту 7777 или недостаточные права доступа к каталогу данных.

### Ошибки хранилища MinIO / S3

- Убедитесь, что `MINIO_ACCESS_KEY` и `MINIO_SECRET_KEY` корректны
- Убедитесь, что контейнер MinIO запущен: `docker compose ps minio`
- Откройте консоль MinIO по адресу `http://localhost:9001` для проверки создания бакета

## Проблемы с развёртыванием Cloudflare

### Ошибки Durable Object

**"Durable Object not found" или ошибки привязки:**

- Запустите `bun run deploy` (никогда не запускайте `wrangler deploy` напрямую) для обеспечения корректности привязок DO
- Проверьте `wrangler.jsonc` на предмет правильных имён и привязок классов DO
- После добавления нового DO необходимо выполнить развёртывание, прежде чем он станет доступным

**Ограничения хранилища DO:**

Cloudflare Durable Objects имеют ограничение 128 КБ на пару ключ-значение. При возникновении ошибок хранилища:

- Убедитесь, что содержимое заметок не превышает лимит (очень большие заметки со многими вложениями)
- Проверьте, что конверты ECIES не дублируются

### Ошибки Worker (ответы 500)

Проверьте журналы Worker:

```bash
bunx wrangler tail
```

Распространённые причины:
- Отсутствующие секреты (используйте `bunx wrangler secret list` для проверки)
- Некорректный формат `ADMIN_PUBKEY` (должен быть 64 шестнадцатеричных символа, без префикса `npub`)
- Ограничение скорости на бесплатном уровне (1000 запросов/минуту на Workers Free)

### Сбой развёртывания с ошибками "Pages deploy"

Никогда не запускайте `wrangler pages deploy` или `wrangler deploy` напрямую. Всегда используйте скрипты корневого `package.json`:

```bash
bun run deploy          # Deploy everything (app + marketing site)
bun run deploy:demo     # Deploy app Worker only
bun run deploy:site     # Deploy marketing site only
```

Запуск `wrangler pages deploy dist` из неправильного каталога развёртывает Vite-сборку приложения на Pages вместо сайта Astro, что приводит к ошибкам 404 на маркетинговом сайте.

## Проблемы с настольным приложением

### Автообновление не работает

Настольное приложение использует Tauri updater для проверки новых версий. Если обновления не обнаруживаются:

- Проверьте подключение к интернету
- Убедитесь, что эндпоинт обновления доступен: `https://github.com/rhonda-rodododo/llamenos-platform/releases/latest/download/latest.json`
- На Linux автообновление AppImage требует наличия прав на запись для файла в его каталоге
- На macOS приложение должно находиться в `/Applications` (не запускаться непосредственно с DMG)

Для ручного обновления загрузите последний выпуск со страницы [Загрузки](/download).

### Разблокировка по PIN не работает

Если ваш PIN отклоняется в настольном приложении:

- Убедитесь, что вводите правильный PIN (восстановление при утере PIN невозможно)
- PIN чувствителен к регистру, если содержит буквы
- Если вы забыли PIN, необходимо повторно ввести ваш nsec для установки нового. Ваши зашифрованные заметки остаются доступными, поскольку они привязаны к вашему идентификатору, а не к PIN
- Tauri Stronghold шифрует ваш nsec с ключом, производным от PIN (PBKDF2). Неверный PIN приводит к некорректной расшифровке, а не к сообщению об ошибке — приложение обнаруживает это, проверяя производный публичный ключ

### Восстановление ключей

Если вы потеряли доступ к устройству:

1. Используйте ваш nsec (который должен храниться в менеджере паролей) для входа на новом устройстве
2. Если вы зарегистрировали WebAuthn-пасскей, можете использовать его на новом устройстве
3. Ваши зашифрованные заметки хранятся на стороне сервера — после входа с тем же идентификатором вы сможете их расшифровать
4. Если вы потеряли и nsec, и пасскей, обратитесь к администратору. Он не может восстановить ваш nsec, но может создать новый идентификатор для вас. Заметки, зашифрованные для вашего старого идентификатора, больше не будут вам доступны

### Приложение не запускается (пустое окно)

- Убедитесь, что ваша система соответствует минимальным требованиям (см. [Загрузки](/download))
- На Linux убедитесь, что WebKitGTK установлен: `sudo apt install libwebkit2gtk-4.1-0` (Debian/Ubuntu) или аналог
- Попробуйте запустить из терминала для просмотра вывода ошибок: `./llamenos` (AppImage) или проверьте системные журналы
- При использовании Wayland попробуйте с `GDK_BACKEND=x11` в качестве резервного варианта

### Конфликт одного экземпляра

Llamenos применяет режим одного экземпляра. Если приложение сообщает, что уже запущено, но вы не можете найти окно:

- Проверьте фоновые процессы: `ps aux | grep llamenos`
- Завершите осиротевшие процессы: `pkill llamenos`
- На Linux проверьте устаревший файл блокировки и удалите его, если приложение аварийно завершилось

## Проблемы с мобильным приложением

### Сбои подготовки

Подробное устранение неполадок при подготовке см. в [Мобильном руководстве](/docs/mobile-guide#устранение-неполадок-мобильного-приложения).

Распространённые причины:
- Истёкший QR-код (токены истекают через 5 минут)
- Отсутствие подключения к интернету на любом устройстве
- Настольное и мобильное приложения запускают разные версии протокола

### Push-уведомления не поступают

- Убедитесь, что разрешения на уведомления предоставлены в настройках ОС
- На Android проверьте, что оптимизация батареи не останавливает приложение в фоне
- На iOS убедитесь, что фоновое обновление приложений включено для Llamenos
- Убедитесь, что у вас активная смена и вы не на перерыве

## Проблемы с телефонией

### Настройка вебхука Twilio

Если звонки не маршрутизируются к волонтёрам:

1. Убедитесь, что URL-адреса вебхуков корректны в консоли Twilio:
   - Голосовой вебхук: `https://your-worker.your-domain.com/telephony/incoming` (POST)
   - Обратный вызов статуса: `https://your-worker.your-domain.com/telephony/status` (POST)
2. Проверьте, что учётные данные Twilio в ваших настройках совпадают с консолью:
   - Account SID
   - Auth Token
   - Номер телефона (должен включать код страны, например `+1234567890`)
3. Проверьте отладчик Twilio на наличие ошибок: [twilio.com/console/debugger](https://www.twilio.com/console/debugger)

### Настройка номера

- Номер телефона должен принадлежать Twilio или быть подтверждённым идентификатором звонящего
- Для локальной разработки используйте Cloudflare Tunnel или ngrok для открытия локального Worker для Twilio
- Убедитесь, что настройка голосовой связи номера указывает на URL вашего вебхука, а не на TwiML Bin по умолчанию

### Звонки соединяются, но нет аудио

- Убедитесь, что медиасерверы провайдера телефонии могут связаться с телефоном волонтёра
- Проверьте наличие проблем NAT/брандмауэра, блокирующих трафик RTP
- При использовании WebRTC убедитесь, что серверы STUN/TURN настроены корректно
- Некоторые VPN блокируют трафик VoIP — попробуйте без VPN

### SMS/WhatsApp сообщения не поступают

- Убедитесь, что URL-адреса вебхуков обмена сообщениями корректно настроены в консоли вашего провайдера
- Для WhatsApp убедитесь, что токен проверки вебхука Meta совпадает с вашими настройками
- Убедитесь, что канал обмена сообщениями включён в **Настройки администратора > Каналы**
- Для Signal убедитесь, что мост signal-cli запущен и настроен для перенаправления на ваш вебхук

## Ошибки криптографии

### Ошибки несовпадения ключей

**"Failed to decrypt" или "Invalid key" при открытии заметок:**

- Обычно это означает, что заметка была зашифрована для другого идентификатора, чем тот, под которым вы вошли
- Убедитесь, что используете правильный nsec (проверьте, что ваш npub в Настройках совпадает с тем, что видит администратор)
- Если вы недавно пересоздали идентификатор, старые заметки, зашифрованные для вашего предыдущего публичного ключа, не будут расшифровываться новым ключом

**"Invalid signature" при входе:**

- nsec может быть повреждён — попробуйте повторно ввести его из менеджера паролей
- Убедитесь, что вставлен полный nsec (начинается с `nsec1`, всего 63 символа)
- Проверьте наличие лишних пробелов или символов новой строки

### Сбои верификации подписи

Если события хаба не проходят верификацию подписи:

- Проверьте синхронизацию системных часов (NTP). Большое расхождение часов может вызывать проблемы с временными метками событий
- Убедитесь, что Nostr-релей не ретранслирует события от неизвестных pubkey
- Перезапустите приложение для повторного получения текущего списка членов хаба

### Ошибки конверта ECIES

**"Failed to unwrap key" при расшифровке заметок:**

- Конверт ECIES мог быть создан с некорректным публичным ключом
- Это может произойти, если администратор добавил волонтёра с опечаткой в pubkey
- Администратор должен проверить публичный ключ волонтёра и при необходимости повторно пригласить его

**"Invalid ciphertext length":**

- Это указывает на повреждение данных, возможно, из-за усечённого сетевого ответа
- Повторите операцию. Если проблема сохраняется, зашифрованные данные могут быть безвозвратно повреждены
- Проверьте наличие прокси или CDN, которые могут усекать тела ответов

### Ошибки хаб-ключа

**"Failed to decrypt hub event":**

- Хаб-ключ мог быть ротирован с момента последнего подключения
- Закройте и снова откройте приложение для получения последнего хаб-ключа
- Если вас недавно удаляли и добавляли обратно в хаб, ключ мог быть ротирован в ваше отсутствие

## Получение помощи

Если ваша проблема не описана здесь:

- Проверьте [GitHub Issues](https://github.com/rhonda-rodododo/llamenos-platform/issues) для получения информации об известных ошибках и обходных путях
- Поищите существующие обращения перед созданием нового
- При сообщении об ошибке укажите: режим развёртывания (Cloudflare/Docker/Kubernetes), платформу (Desktop/Mobile) и любые сообщения об ошибках из консоли браузера или терминала
